# 二、数据分析与数仓索引

这一组是旧 Warp skills 没有覆盖的新方向：让 agent 在数据分析中留下可复现的 SQL、可视化和模型索引，而不是只给一次性答案。


### `analysis-artifacts`

用途：做 BigQuery 深度分析时，持续生成可复现的分析目录、SQL 查询、Python 可视化和总结文档。

```text
+------------------------------------------------------------+
| 触发：用户要求 deep dive / analysis / investigation，且问题   |
|       不是简单 lookup，需要探索性 BigQuery 查询。             |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 1. 初始化分析目录                                            |
| analyses/<date-or-topic>/                                    |
| - README.md                                                  |
| - assets/queries/                                            |
| - assets/visualizations/                                     |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 2. 先写计划并等用户确认                                      |
| - 计划映射到子问题/子区域。                                  |
| - 即使用户没要求 plan，也要先计划。                           |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 3. 设置 README 框架                                          |
| - title / author / date                                      |
| - Problem Statement                                          |
| - Cohorts Definition                                         |
| - 后续补 TL;DR 和 Key Takeaways                              |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 4. 明确定义 cohorts                                          |
| 控制混杂因素：                                               |
| - signup / activation time                                   |
| - plan / subscription tier                                   |
| - observation window length                                  |
| - usage propensity metrics                                   |
| 后续所有 query 必须遵守同一 cohort 定义。                    |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 5. 按分析步骤生成 artifacts                                  |
| - 重要 BigQuery query 保存为 .sql，带目的注释。              |
| - 可视化用 Python script 生成并保存 image。                  |
| - 表格保存为 .csv。                                          |
| - 跳过 trivial/one-off lookup。                              |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 判断：方法是否重做？                                         |
|                                                            |
| 是：覆盖相关 .sql、script、image/csv，并告知用户。            |
| 否：继续积累 artifacts。                                     |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 6. 写完整 README                                             |
| - 按子问题总结。                                             |
| - 嵌入 visualizations。                                      |
| - CSV 表转 Markdown 表。                                     |
| - 每段加对应 query 小链接。                                  |
| - TL;DR 靠前，Key Takeaways 结尾。                           |
+------------------------------------------------------------+
```

解释：原 skill：[analysis-artifacts](https://github.com/warpdotdev/oz-skills/blob/main/.agents/skills/analysis-artifacts/SKILL.md)。

要点：
- 新增的是“可复现分析产物”纪律：每个重要结论都能回到 SQL、可视化脚本或 CSV，而不是只保留聊天结论。
- cohort 定义是架构核心，因为它决定后续查询能否比较同一类对象、同一观察窗口和同一产品语义。
- 容易踩的坑：先查一堆数再补计划、cohort 在不同 query 中漂移、图表没有脚本来源、或 README 缺少 query 反链。

### `dbt-model-index`

用途：在 dbt 驱动的数据仓库中提供模型索引，帮助 agent 把模糊数据问题映射到正确 BigQuery 表。

```text
+------------------------------------------------------------+
| 触发：需要写 BigQuery SQL，但用户没有明确指定模型/表，或问题 |
|       需要从业务语义解析到 dbt 模型。                        |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 1. 按领域组织模型                                            |
| 常见 domain：                                                |
| - Users & Identity                                           |
| - Activity & Engagement                                      |
| - Revenue & Subscriptions                                    |
| - Events & Telemetry                                         |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 2. 为每个模型记录最小上下文                                  |
| - table/model name                                           |
| - grain：一行代表什么实体/时间粒度                           |
| - 1-2 句说明内容                                             |
| - Useful for：适合回答的问题                                 |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 3. 标注查询关键约束                                          |
| - join keys                                                  |
| - standard filters                                           |
| - partition fields                                           |
| - 常见 join pattern                                          |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 4. 使用索引写查询                                            |
| - 先根据问题选择模型，而不是凭表名猜。                       |
| - 遇到模糊业务概念时回到 domain 和 grain。                   |
| - 不能确认模型含义时继续查 dbt/docs，而不是写 SQL。           |
+------------------------------------------------------------+
```

解释：原 skill：[dbt-model-index](https://github.com/warpdotdev/oz-skills/blob/main/.agents/skills/dbt-model-index/SKILL.md)。

要点：
- 这是一个可定制模板，不是现成数据字典；价值在于把模型的 grain、用途和 join 约束写给 agent。
- 它和 `analysis-artifacts` 互补：先用 index 选对表，再用 artifacts 留下可复现分析。
- 容易踩的坑：只列 table name 不写 grain、漏掉标准 filters、或让 agent 在不理解模型语义时直接拼 SQL。
