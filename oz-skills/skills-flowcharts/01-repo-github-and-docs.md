# 一、代码仓库、GitHub 与文档工作流

这一组 skill 把旧 Warp 仓库里的 PR、CI 和 Issue 经验抽象成更通用的 GitHub/`gh` 工作流，同时新增了文档自动更新和只读调查模式。


### `create-pull-request`

用途：按项目约定创建 GitHub PR，覆盖前置检查、上下文收集、标题/正文生成、分支推送和 PR 创建。

```text
+------------------------------------------------------------+
| 触发：用户要求 create/open PR、submit for review，或当前改动 |
|       需要进入 GitHub review。                              |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 1. 前置检查                                                  |
| - gh --version                                               |
| - gh auth status                                             |
| - 查找 review / CI / testing 相关 skills，必要时先执行       |
| - git status，确认未提交改动如何处理                         |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 2. 判断当前分支是否已有 PR                                   |
| gh pr list --head $(git branch --show-current)               |
|                                                            |
| 已有 PR：展示编号/标题/URL，询问查看、更新或新建策略。       |
| 无 PR：继续创建新 PR。                                      |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 3. 收集 Git 上下文                                           |
| - 当前 branch，不能直接在 main/master 上开。                 |
| - origin HEAD branch 作为 base。                             |
| - git log origin/main..HEAD 理解提交范围。                   |
| - git diff origin/main..HEAD --stat 理解文件和改动类型。     |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 4. 生成 PR 信息                                              |
| - 从 commit、branch、diff 提取 issue number。                |
| - 判断 type：bug/feature/breaking/refactor/docs/workflow。   |
| - 根据项目习惯决定 conventional commits 标题或描述性标题。   |
| - 读取 .github/pull_request_template.md 并填充正文。         |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 判断：是否用 draft？                                         |
|                                                            |
| draft，如果：WIP、测试失败、需要架构反馈、只想触发 CI。       |
| ready，如果：改动完整、测试通过、准备正式 review。            |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 5. 推送并创建 PR                                             |
| - git push origin HEAD                                      |
| - 必要时 --force-with-lease                                 |
| - gh pr create --title ... --body ... --base ... [--draft]  |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 6. 创建后核验                                                |
| - gh pr view --web                                           |
| - 检查标题、正文、链接、diff 和 CI 初始状态。                |
| - 如需要，补 reviewer 或 label。                             |
+------------------------------------------------------------+
```

解释：原 skill：[create-pull-request](https://github.com/warpdotdev/oz-skills/blob/main/.agents/skills/create-pull-request/SKILL.md)。

要点：
- 相比旧 `create-pr`，它不再绑定 Warp 的 Linear、`origin/master` 和 PR template 约定，而是从仓库本身推断 base、commit style、issue reference 和模板。
- 新增“相关 skill 前置检查”，也就是在开 PR 前先考虑 review、CI、testing、UI guideline 等工作流是否已满足。
- 容易踩的坑：标题过于泛化、已有 PR 时重复创建、未认证 `gh`、没有读 PR template、或者把明显未完成的改动开成 ready PR。

### `ci-fix`

用途：诊断并修复 GitHub Actions CI 失败，强调最小修复和专门的 fix branch。

```text
+------------------------------------------------------------+
| 触发：CI red / failing / broken，用户要求诊断或修复 checks。 |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 1. 确认 gh 可用并已登录                                      |
| gh auth status                                               |
| 未登录：要求用户先 gh auth login。                           |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 2. 定位失败 run                                              |
| PR branch：gh pr view --json statusCheckRollup               |
| branch/run id：gh run list / gh run view --verbose           |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 3. 抽取失败日志                                              |
| - gh run view <run-id> --log-failed                          |
| - 必要时 gh run view --log --job <job-id>                    |
| - 必要时 gh run download 到 artifacts                        |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 4. 归因                                                      |
| - build/compile：依赖、类型、语法                             |
| - test：断言、超时、flake                                    |
| - lint/format：style、unused import                          |
| - environment：secret、permission、resource                   |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 5. 最小修复                                                  |
| - 只改失败原因相关代码。                                     |
| - workflow 改动不扩大 token 权限。                           |
| - 确定性代码修复优先于 CI plumbing。                         |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 6. 推送 fix branch 并验证                                    |
| - ci-fix/<original-branch>                                   |
| - commit 描述具体 job。                                      |
| - push 后 gh run watch / gh pr checks。                      |
+------------------------------------------------------------+
```

解释：原 skill：[ci-fix](https://github.com/warpdotdev/oz-skills/blob/main/.agents/skills/ci-fix/SKILL.md)。

要点：
- 它把旧 `diagnose-ci-failures` 和 `fix-errors` 合并成一个可执行闭环：找 run、读日志、修根因、推分支、观察验证。
- 和旧 Warp 流程不同，它明确“不创建 PR”，只推 dedicated fix branch。
- 容易踩的坑：还没读 failed step 日志就猜原因、用宽泛 workflow 改动掩盖代码问题、扩大 GitHub token 权限、或顺手做无关 refactor。

### `docs-update`

用途：根据近期代码变化更新用户可见文档，支持 dry-run 报告和自动化执行。

```text
+------------------------------------------------------------+
| 触发：用户要求更新文档、审查文档、处理定时文档任务，或分析    |
|       近期 commits 是否需要 docs。                          |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 1. 找出重要代码变化                                          |
| - default branch                                             |
| - 最近 commits，默认 24 小时或用户指定范围                   |
| - 检查 diff 的用户可见影响                                   |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 判断：是否值得更新 docs？                                    |
|                                                            |
| 需要，如果：新功能、API/CLI/config 变化、breaking change、    |
| 用户可见行为变化。                                           |
|                                                            |
| 跳过，如果：内部 refactor、test-only、小 typo、小 bugfix、    |
| 无用户影响的性能优化。                                       |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 2. 分析文档上下文                                            |
| - 定位 docs 目录或独立 docs repo。                           |
| - 判断 Mintlify / Docusaurus / GitBook / Fumadocs / Markdown。 |
| - 阅读现有文档风格、术语、代码示例和平台语法。               |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 3. 规划文档变更                                              |
| - 将代码变化映射到新增、修改或补充文档。                     |
| - 优先用户任务，不写实现细节。                               |
| - 尽量 additive，保留仍然准确的原文。                        |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 判断：dry-run 还是执行？                                     |
|                                                            |
| dry-run：输出会改哪些文件、改什么、为什么。                  |
| 执行：建 docs/auto-update-* 分支，改文档、提交、推送、开 PR。 |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 4. 多仓库/边界处理                                           |
| - 代码仓库和 docs 仓库分离时，在 PR 描述里引用源 commits。   |
| - 无重要变化时说明不需要更新。                               |
| - 文档平台不明确时使用标准 Markdown。                        |
+------------------------------------------------------------+
```

解释：原 skill：[docs-update](https://github.com/warpdotdev/oz-skills/blob/main/.agents/skills/docs-update/SKILL.md)。

要点：
- 这是旧 Warp skills 没覆盖的新自动化方向：代码变化触发文档同步。
- 核心 tradeoff 是保守：宁可跳过，也不要为了每个内部变化污染用户文档。
- 容易踩的坑：只看一个文件就改文档、忽略文档平台语法、把实现细节写成用户指南、或没把 source commits 和 docs PR 对上。

### `github-bug-report-triage`

用途：判断 GitHub bug report 是否有足够信息进入调查。

```text
+------------------------------------------------------------+
| 触发：需要评估 GitHub issue 中的 bug report 是否 actionable。 |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 1. 查找 issue template                                       |
| 优先：.github/ISSUE_TEMPLATE/bug_report.{md,yml}             |
| 其次：.github/ISSUE_TEMPLATE/*.md、ISSUE_TEMPLATE.md         |
| 没有：使用 skill 自带 references/ISSUE_TEMPLATE.md。          |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 2. 判断 issue 类型                                           |
| - 是 bug report：继续评估。                                  |
| - 是 feature request 或 question：不按 bug 标准评估。         |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 3. 评估最低可行动信息                                        |
| - 问题描述是否清楚：broken/expected/actual/impact。           |
| - 是否可复现：步骤、sandbox 链接或最小代码示例。             |
| - 是否有关键上下文：错误信息、环境、版本等。                 |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 判断：是否 actionable？                                      |
|                                                            |
| ready：问题清楚，并有复现路径或足够 debug 上下文。            |
| needs info：描述含糊、无复现、期望不明、缺关键错误/环境。     |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 4. 输出反馈                                                  |
| ready：确认可以调查。                                        |
| needs info：具体说明缺什么，避免机械要求所有模板字段。       |
+------------------------------------------------------------+
```

解释：原 skill：[github-bug-report-triage](https://github.com/warpdotdev/oz-skills/blob/main/.agents/skills/github-bug-report-triage/SKILL.md)。

要点：
- 它是旧 issue triage 的 bug-only 细化版本，关注“能不能开始调查”而不是分类、标签或优先级全流程。
- 判断标准是合理而非机械：明显 bug 不必强迫填写所有字段；模糊 bug 要指出最小缺失信息。
- 容易踩的坑：把 feature/question 当 bug 评估、强制模板每项都填、或反馈只说“needs more info”但不告诉 reporter 缺什么。

### `github-issue-dedupe`

用途：用多策略 GitHub 搜索识别重复 issue，并且只在高置信度时评论。

```text
+------------------------------------------------------------+
| 触发：用户要求找 duplicate、similar issues，或设置自动查重。 |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 1. 提取目标 issue 上下文                                     |
| - issue number/title/body                                    |
| - error messages / stack traces                              |
| - keywords / technical terms                                 |
| - symptoms / behavior descriptions                           |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 2. 多策略搜索                                                |
| - title/body keyword search                                  |
| - error message exact/near search                            |
| - symptom search                                             |
| - 通过 gh issue list / gh search 拉候选                      |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 3. 候选逐个检查                                              |
| 对每个候选 gh issue view：                                   |
| - root cause 是否相同？                                      |
| - symptoms 是否相同？                                        |
| - error messages 是否一致或高度相似？                        |
| - affected component 是否相同？                              |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 判断：置信度是否足够高？                                     |
|                                                            |
| 高置信：相同错误、相同复现、同一 root cause、互相引用。       |
| 低置信：症状类似但原因不同，或只是同组件不同问题。            |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 4. 报告/评论                                                 |
| - 90%+ certainty 才评论。                                    |
| - 格式：This is potentially a duplicate of #123 and #456.    |
| - 不确定时不评论。                                           |
+------------------------------------------------------------+
```

解释：原 skill：[github-issue-dedupe](https://github.com/warpdotdev/oz-skills/blob/main/.agents/skills/github-issue-dedupe/SKILL.md)。

要点：
- 相比旧 `dedupe-issue-local`，这里更强调通用 GitHub 搜索和“高置信才写评论”的安全边界。
- Dedupe 的核心不是相似度，而是同一 root cause；症状相似但原因不同不应合并。
- 容易踩的坑：看到同组件就判重复、没读候选全文、低置信也自动评论、或评论格式不统一。

### `slack-qa-investigate`

用途：只读调查仓库问题，给出带证据的回答，不修改任何文件。

```text
+------------------------------------------------------------+
| 触发：用户提出需要代码库/文档调查才能回答的问题。             |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 1. 明确问题范围                                              |
| - 理解用户真正问什么。                                       |
| - 范围不清时先澄清。                                         |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 2. 广泛搜索                                                  |
| - 语义搜索 / grep / 文件名搜索。                             |
| - 找入口、调用点、配置、测试、文档。                         |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 3. 深读并追踪                                                |
| - 读真实代码，不只看文件名。                                  |
| - 顺着 imports、函数调用、README 链接继续追。                |
| - 外部库/API 需要时查官方文档。                              |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 4. 严格只读                                                  |
| - 不创建、编辑、删除文件。                                    |
| - 不运行有副作用命令：install、commit、write、delete。        |
| - 用户要求改代码时拒绝，并说明只能调查/解释。                |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 5. 合成回答                                                  |
| - 引用具体文件/行号。                                        |
| - 区分代码事实、文档说法和推断。                             |
| - 找不到就说找不到，不猜。                                   |
+------------------------------------------------------------+
```

解释：原 skill：[slack-qa-investigate](https://github.com/warpdotdev/oz-skills/blob/main/.agents/skills/slack-qa-investigate/SKILL.md)。

要点：
- 这是新增的“研究型只读 agent”模式，特别适合 Slack 问答或代码库解释。
- 它的架构约束是能力边界：为避免副作用，宁可不提供 patch，也要保持调查结果可信。
- 容易踩的坑：从记忆回答、没追链接、把推断当事实、或者在“只读模式”里顺手改文件。
