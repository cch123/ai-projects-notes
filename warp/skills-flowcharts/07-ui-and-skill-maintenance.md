# 七、UI 规范与 Skill 维护

用于约束 Warp UI 实现风格，并把可复用工作流沉淀成新的 skill 或更新现有 skill。


### `warp-ui-guidelines`

用途：做任何 Warp UI 改动前读取并应用 UI 编码准则。

```text
+------------------------------------------------------------+
| 触发：任务涉及 Warp UI 代码、View、button、theme、layout 或  |
|       任何用户可见界面。                                    |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 1. 开始实现前先读 guidelines                                |
| 目的：让实现遵循现有设计系统，避免 review 后才发现 one-off。 |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 判断：是否涉及 button 样式？                                |
|                                                            |
| 涉及，如果：                                                 |
| - 新增/修改按钮。                                            |
| - 调整按钮颜色、文字颜色、hover/active/disabled 状态。       |
| - 新建 ActionButtonTheme 或 ui_components button theme。     |
|                                                            |
| 不涉及：继续应用其他 UI repo patterns。                     |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 2. button theme 选择                                         |
| 优先使用已有 theme：                                         |
| - PrimaryTheme                                               |
| - SecondaryTheme                                             |
| - NakedTheme                                                 |
| - DangerPrimaryTheme                                         |
| - 以及 app/src/view_components/action_button.rs 和           |
|   crates/ui_components/src/button/themes.rs 中已有角色主题。 |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 判断：是否可以新建/修改 theme？                             |
|                                                            |
| 不应新建 one-off theme，如果：                              |
| - 只是想让某个 feature 的按钮颜色略有不同。                  |
| - 新 theme 仅 delegate existing theme 并改一个方法。         |
| - theme 名称按 feature/view 命名，例如 FooPrimaryTheme。     |
|                                                            |
| 不应修改共享 theme，如果：                                  |
| - 用户没有明确要求改设计系统组件本身。                       |
| - 改动会影响全 app 的 Primary/Secondary 等按钮。             |
|                                                            |
| 可以提出设计系统改动，如果：                                |
| - 现有 theme 确实无法表达所需设计角色。                      |
| - 用户明确确认要改共享组件。                                |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 3. 禁止/警惕信号                                             |
| 标记自己正在走偏，如果：                                     |
| - hard-code ColorU::new(...)。                               |
| - 不用 appearance.theme() accessors。                        |
| - 设置 should_opt_out_of_contrast_adjustment = true          |
|   只是为了强制 label color。                                |
| - 修改 PrimaryTheme/SecondaryTheme 来修单个页面。            |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 4. 如果 theme 看起来不对                                     |
| 先判断：                                                     |
| - 是否选错了已有 theme？                                     |
| - 是否应该接受现有设计系统结果？                             |
| - 是否需要向用户确认设计系统本身变更？                       |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 验证                                                        |
| - UI 使用共享组件/主题。                                     |
| - 没有局部硬编码颜色。                                       |
| - 没有未经确认修改共享 theme。                               |
| - 用户可见 UI 改动按风险做 layout/screenshot/manual 验证。   |
+------------------------------------------------------------+
```

解释：原 skill：[warp-ui-guidelines](https://github.com/warpdotdev/warp/blob/master/.agents/skills/warp-ui-guidelines/SKILL.md)。

要点：
- 适用场景：任何 Warp UI 改动，尤其是按钮、theme、View、布局或用户可见状态。
- 核心判断：优先复用现有设计系统和 `ActionButtonTheme`/button themes。只有用户明确确认设计系统组件本身需要变化时，才修改共享 theme。
- 关键产物：使用现有 theme 的 UI 改动、没有 hard-coded `ColorU::new(...)` 的局部样式、必要的 layout/visual 验证。
- 容易踩的坑：为了单个页面新建 feature-specific theme、修改 `PrimaryTheme`/`SecondaryTheme` 影响全局、或用 contrast opt-out 强行得到某个 label color。

### `update-skill`

用途：创建或更新仓库中的 `SKILL.md`。

```text
+------------------------------------------------------------+
| 触发：用户要求新增 skill、修改 skill、整理 skill 指南，或    |
|       需要把重复流程沉淀成 skill。                          |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 1. 创建/定位目录                                             |
| .agents/skills/<skill-name>/SKILL.md                        |
| skill-name 要 kebab-case。                                  |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 2. 写 YAML frontmatter                                      |
| 必填：                                                       |
| ---                                                         |
| name: kebab-case-identifier                                 |
| description: Action verb + what + when to use               |
| ---                                                         |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 判断：description 是否合格？                                |
|                                                            |
| 合格，如果：                                                 |
| - 具体说明 skill 做什么。                                    |
| - 说明什么时候用。                                          |
| - 包含发现该 skill 所需关键词。                              |
| - 第三人称、动作动词开头。                                  |
|                                                            |
| 不合格，如果：                                               |
| - "Helps with code" 这类泛泛描述。                          |
| - 只写能力，不写触发场景。                                  |
| - 使用第一人称。                                            |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 3. 组织正文                                                  |
| 常见结构：                                                   |
| - Title                                                      |
| - Overview                                                   |
| - Main workflow / Steps                                      |
| - Best Practices                                             |
| - Examples / Reference PRs                                   |
| 结构可按 skill 复杂度调整，不必每节都写。                    |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 判断：是否拆 references/？                                  |
|                                                            |
| 保持单文件，如果：                                           |
| - SKILL.md <= 约 200 行。                                    |
| - workflow 简单。                                            |
| - 没有大量 schema/example/reference。                        |
|                                                            |
| 拆 references/，如果：                                      |
| - SKILL.md 接近或超过 200 行。                               |
| - 覆盖多个可独立加载的 workflow。                            |
| - 详细参考资料会淹没主流程。                                |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 4. 写 main content                                           |
| 好内容：                                                     |
| - 可执行步骤。                                               |
| - 关键判断依据。                                             |
| - 具体命令/文件/产物。                                       |
| - 反模式和注意事项。                                        |
|                                                            |
| 避免：                                                       |
| - 空泛原则。                                                 |
| - 大段背景但没有操作路径。                                  |
| - 不必要地复制外部文档。                                     |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 5. 可选验证                                                  |
| skills-ref validate ./my-skill                              |
| 如未安装，可根据需要查官方 reference。                       |
+------------------------------------------------------------+
```

解释：原 skill：[update-skill](https://github.com/warpdotdev/warp/blob/master/.agents/skills/update-skill/SKILL.md)。

要点：
- 适用场景：新增 skill、更新现有 skill，或把重复工程流程沉淀成 agent 可复用的本地说明。
- 核心判断：skill 要服务发现和执行。frontmatter `description` 必须清楚写出“做什么”和“什么时候用”；正文应给步骤、判断依据、文件/命令/产物，而不是泛泛原则。
- 关键产物：`.agents/skills/<name>/SKILL.md`、有效 YAML frontmatter、可执行 workflow、必要时拆出的 `references/`。
- 容易踩的坑：description 太泛、正文只写背景不写操作、单个 `SKILL.md` 超长却不拆 reference，或把大量外部文档逐字复制进 skill。
