# 一、规格驱动与实现规划

用于把模糊功能变成可实现、可验证、可 review 的工作流。先定义用户行为，再写技术方案，最后实现并保持 specs 与代码同步。


### `spec-driven-implementation`

用途：为较大或高风险功能执行 spec-first 工作流。

```text
+------------------------------------------------------------+
| 触发：开始显著功能、用户要求 product/tech spec，或任务复杂到 |
|       先写规格会明显提高实现和 review 质量。                |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 判断：是否需要 specs？                                      |
|                                                            |
| 需要 specs，如果满足任一条件：                              |
| - 产品行为不明确，存在多个合理解释。                         |
| - 预计实现约 1k+ LOC。                                      |
| - 跨多个模块、子系统或 ownership 边界。                      |
| - 行为改变风险高，回归成本大。                               |
| - agent 执行质量依赖更清晰输入。                             |
| - reviewer 需要先确认行为或架构，再看代码。                  |
|                                                            |
| 通常不需要 specs，如果属于：                                |
| - 小型局部 bug fix。                                        |
| - 直接 refactor。                                            |
| - 窄 UI tweak，行为和视觉都明确。                            |
| - 单文件/单模块、低风险、验证路径清楚。                      |
+-----------------------------+------------------------------+
                 |                              |
                 v                              v
+--------------------------------+  +--------------------------------+
| 不需要 specs                   |  | 需要 specs                     |
| 直接实现 + targeted 验证       |  | 进入 specs workflow            |
+----------------+---------------+  +---------------+----------------+
                 |                                  |
                 |                                  v
                 |                  +--------------------------------+
                 |                  | 1. 准备 specs/<ticket>/        |
                 |                  | - Linear ticket 优先            |
                 |                  | - 无 ticket 时明确 feature id   |
                 |                  | - specs/ 直接子目录只放 id      |
                 |                  +---------------+----------------+
                 |                                  |
                 |                                  v
                 |                  +--------------------------------+
                 |                  | 2. PRODUCT.md                  |
                 |                  | 使用 write-product-spec         |
                 |                  | 描述用户行为、不变量、边界状态 |
                 |                  +---------------+----------------+
                 |                                  |
                 |                                  v
                 |                  +--------------------------------+
                 |                  | 判断：是否需要 TECH.md？       |
                 |                  |                                |
                 |                  | 需要，如果：                   |
                 |                  | - 多模块实现                   |
                 |                  | - 有架构或扩展性取舍           |
                 |                  | - reviewer 需要计划            |
                 |                  |                                |
                 |                  | 可跳过，如果：                 |
                 |                  | - 纯 UI 小改                   |
                 |                  | - 实现方式明显                 |
                 |                  +---------------+----------------+
                 |                                  |
                 |                       +----------+----------+
                 |                       v                     v
                 |       +----------------------------+  +----------------------+
                 |       | 写 TECH.md                  |  | 跳过 TECH.md         |
                 |       | write-tech-spec             |  | 只保留 PRODUCT.md    |
                 |       +-------------+--------------+  +----------+-----------+
                 |                     |                            |
                 |                     +-------------+--------------+
                 |                                   |
                 v                                   v
+------------------------------------------------------------+
| 3. specs review / approval                                  |
| - 行为足够清楚。                                             |
| - 技术计划足够可执行。                                       |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 4. implement-specs                                           |
| - 代码、测试、spec 更新保持同 PR 同步。                      |
| - 实现变化时更新 PRODUCT.md / TECH.md。                      |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 5. 验证                                                      |
| - tests/artifacts 映射回 PRODUCT behavior。                  |
| - TECH validation plan 被执行或明确说明未执行原因。          |
+------------------------------------------------------------+
```

解释：原 skill：[spec-driven-implementation](https://github.com/warpdotdev/warp/blob/master/.agents/skills/spec-driven-implementation/SKILL.md)。

要点：
- 适用场景：功能较大、行为有歧义、预计跨模块或高风险，直接实现会让 agent 和 reviewer 都缺少稳定输入。
- 核心判断：不是所有改动都写 specs。小修、局部 refactor、窄 UI tweak 可以跳过；产品行为不清、约 1k+ LOC、跨子系统、回归成本高时应进入 PRODUCT -> TECH -> implement flow。
- 关键产物：`specs/<id>/PRODUCT.md`、必要时 `TECH.md`、批准后的实现和验证，以及实现期间持续更新的 specs。
- 容易踩的坑：为小改动过度流程化，或相反，在复杂功能上跳过规格，导致实现阶段不断重新解释需求。

### `write-product-spec`

用途：为重要用户可见功能写 `specs/<id>/PRODUCT.md`。

```text
+------------------------------------------------------------+
| 触发：用户要求 PRD/产品规格/期望行为文档，或功能行为复杂到   |
|       先写 PRODUCT.md 能减少歧义。                          |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 1. 确认 spec id                                              |
| 优先顺序：                                                   |
| - Linear ticket，例如 specs/APP-1234/PRODUCT.md              |
| - GitHub issue，例如 specs/gh-4567/PRODUCT.md                |
| - 短 kebab-case feature name                                 |
|                                                            |
| 规则：                                                       |
| - specs/ 直接子目录只放 id。                                 |
| - 不创建 engineer-named subdirectories。                     |
| - 只有用户明确要求时才新建 Linear/GitHub issue。             |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 2. 收集必要上下文                                            |
| - feature summary                                            |
| - target users / consumers                                   |
| - key behaviors                                              |
| - edge cases                                                 |
| - validation expectations for future TECH.md                 |
| 缺关键信息时问用户，不猜。                                   |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 判断：是否需要问 Figma？                                    |
|                                                            |
| 需要问，如果：                                               |
| - UI 或 interaction design 会影响 Behavior。                  |
| - visual states、spacing、edge layouts 可能已有设计答案。    |
|                                                            |
| 不需要问，如果：                                             |
| - 纯 backend/data model/API/CLI，无视觉 surface。            |
|                                                            |
| 如果用户提供 Figma：写入 Figma section 或 Behavior 附近。    |
| 如果确认没有 mock：写 Figma: none provided。                 |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 3. 写 Summary                                                |
| - 1-3 句。                                                   |
| - 说明功能和期望结果。                                       |
| - 不堆实现细节。                                             |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 判断：是否写 optional sections？                            |
|                                                            |
| Problem，如果：                                               |
| - 动机不从 Summary 自明。                                    |
|                                                            |
| Goals / Non-goals，如果：                                    |
| - scope 有歧义或曾被争论。                                   |
|                                                            |
| Figma，如果：                                                |
| - 设计上下文对 UI 行为有意义。                               |
|                                                            |
| Open questions，如果：                                       |
| - 多个 unresolved questions 值得集中列出。                   |
| - 单个问题优先 inline 到相关 Behavior。                      |
|                                                            |
| 不写空 section，不写 None placeholder。                      |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 4. 写 Behavior                                               |
| Behavior 是核心，必须编号、可测试、用户视角。                |
| 至少覆盖：                                                   |
| - default behavior / happy path                              |
| - all user-visible states and transitions                    |
| - all inputs and responses                                   |
| - empty/loading/error/pending/cancellation                   |
| - permissions/offline/timeouts/races/stale data              |
| - focus loss / multiple instances / adjacent features        |
| - keyboard/accessibility/focus expectations                  |
| - invariants that must not regress                           |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 禁止内容                                                     |
| - internal types                                             |
| - module boundaries                                          |
| - state layout                                               |
| - data flow                                                  |
| - algorithms                                                 |
| - Validation / Success criteria / Testing sections           |
| 验证计划属于 TECH.md。                                      |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 完成标准                                                     |
| - Behavior 足够让 TECH.md 作者无需猜产品意图。               |
| - 每个重要状态和边界都有明确期望。                           |
| - 规格描述消费者视角，而不是实现者视角。                     |
+------------------------------------------------------------+
```

解释：原 skill：[write-product-spec](https://github.com/warpdotdev/warp/blob/master/.agents/skills/write-product-spec/SKILL.md)。

要点：
- 适用场景：需要 PRD/产品规格，或功能行为复杂到需要先把用户视角、不变量和边界状态写清楚。
- 核心判断：PRODUCT.md 只描述消费者可观察行为，不写内部类型、模块、状态布局或算法。Behavior section 是核心，应该编号、可测试、覆盖 happy path 和 edge cases。
- 关键产物：`specs/<id>/PRODUCT.md`，包含 Summary、必要的 Problem/Goals/Figma/Open questions，以及详尽的 Behavior invariants。
- 容易踩的坑：把实现方案写进产品规格、跳过 Figma 询问、写空的模板 section，或把 Validation/Testing 放进 PRODUCT.md 而不是 TECH.md。

### `write-tech-spec`

用途：为重要功能写 `specs/<id>/TECH.md`，把产品行为转成贴合现有代码的实现计划。

```text
+------------------------------------------------------------+
| 触发：用户要求技术规格、实现计划、架构文档，或功能跨模块/有  |
|       明显实现取舍。                                        |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 判断：是否适合写 TECH.md？                                  |
|                                                            |
| 适合，如果：                                                 |
| - implementation spans multiple modules。                    |
| - architecture/extensibility matters。                       |
| - 有多个合理方案，需要记录 tradeoffs。                       |
| - reviewer 先看计划比直接看 diff 更有效。                    |
|                                                            |
| 通常不需要，如果：                                           |
| - 纯 UI 小改。                                               |
| - straightforward fix。                                      |
| - 单文件、实现路径明显。                                     |
|                                                            |
| 如果实现仍太不确定：                                        |
| - 可以先做 e2e prototype，再基于发现写 TECH.md。             |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 1. 确认 spec id                                              |
| - 与 sibling PRODUCT.md 使用同一 <id>。                      |
| - 写到 specs/<id>/TECH.md。                                  |
| - 无 id 时问用户 feature name，不擅自创建不清楚目录。        |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 2. 先读 PRODUCT.md                                           |
| - TECH.md 引用 PRODUCT.md 的行为编号。                       |
| - 不重复展开产品行为。                                       |
| - 如果没有 PRODUCT.md，至少确认产品意图来源。                |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 3. 研究当前代码                                              |
| 必须实际 inspect：                                           |
| - relevant files                                             |
| - types/state                                                |
| - data flow                                                  |
| - ownership boundaries                                       |
| - existing patterns                                          |
| - risky call paths                                           |
| 不能凭空猜架构。                                             |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 4. 写 Context                                                |
| 内容：                                                       |
| - 要构建什么。                                               |
| - 当前系统在相关区域如何工作。                               |
| - 最相关文件和行号。                                         |
| - 当前限制和约束。                                           |
|                                                            |
| 不要写成泛泛背景；每段都应帮助理解实现计划。                 |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 5. 写 Proposed changes                                       |
| 必须具体说明：                                               |
| - 改哪些 modules/files。                                     |
| - 新增哪些 types/APIs/state。                                |
| - data flow 怎么走。                                         |
| - ownership boundaries 如何保持清楚。                        |
| - 设计如何遵循现有 patterns。                                |
| - 有多个方案时说明 tradeoffs。                               |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 6. 写 Testing and validation                                 |
| - 映射 PRODUCT.md Behavior 编号。                            |
| - 每个重要 invariant 对应 unit/integration/manual 验证。     |
| - 包括 screenshots/videos 等 UI 验证 artifact，如有必要。    |
| - 这里是 validation 所在位置，PRODUCT.md 不写验证 section。  |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 判断：是否需要 optional sections？                          |
|                                                            |
| End-to-end flow，如果：                                      |
| - 追踪跨层路径比列表更清楚。                                 |
|                                                            |
| Diagram，如果：                                              |
| - 数据流、状态机、跨层 sequence 用图更快解释。               |
|                                                            |
| Risks and mitigations，如果：                                |
| - 存在真实 failure modes、migration、rollout hazards。       |
|                                                            |
| Parallelization，如果：                                      |
| - 工作可安全拆给多个 agent，且拆分边界不明显。               |
|                                                            |
| Follow-ups，如果：                                           |
| - 有明确延后清理或未来工作。                                 |
|                                                            |
| 不写空 section。                                             |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 7. 保持 TECH.md 当前                                         |
| 实现期间更新 TECH.md，如果：                                |
| - module boundaries 改变。                                   |
| - implementation sequencing 改变。                           |
| - risks/dependencies/rollout assumptions 改变。              |
| - testing/validation plan 改变。                             |
+------------------------------------------------------------+
```

解释：原 skill：[write-tech-spec](https://github.com/warpdotdev/warp/blob/master/.agents/skills/write-tech-spec/SKILL.md)。

要点：
- 适用场景：实现跨多个模块、有架构取舍、reviewer 需要先理解计划，或用户明确要求技术规格/实现计划。
- 核心判断：TECH.md 必须先研究真实代码，并引用相关文件/行号；它应把 PRODUCT.md 的行为映射到模块、类型、状态、数据流和验证计划，而不是重复产品描述。
- 关键产物：`specs/<id>/TECH.md`，包含 Context、Proposed changes、Testing and validation，以及必要的 Risks/Diagram/Parallelization/Follow-ups。
- 容易踩的坑：没有读代码就写泛泛架构、验证计划没有映射 PRODUCT Behavior 编号、实现变化后不更新 TECH.md。

### `implement-specs`

用途：从已批准的 PRODUCT.md / TECH.md 实现功能，并保持 specs 与代码同步。

```text
+------------------------------------------------------------+
| 触发：产品和技术规格已经足够批准，用户要求开始实现。         |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 1. 检查前置条件                                              |
| - specs/<id>/PRODUCT.md 存在。                               |
| - 如果功能需要 TECH.md，则 specs/<id>/TECH.md 存在。          |
| - 用户或 review 已允许开始实现。                             |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 2. 读取 specs                                                |
| PRODUCT.md：用户可见行为、不变量、边界状态的 source of truth |
| TECH.md：模块、状态、数据流、实现顺序、验证计划的 source。   |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 判断：是否需要辅助跟踪文档？                                |
|                                                            |
| 可提议 PROJECT_LOG.md，如果：                               |
| - 功能跨很多步骤。                                          |
| - 需要记录探索路径和当前状态，方便未来 agent 接手。          |
|                                                            |
| 可提议 DECISIONS.md，如果：                                 |
| - 产品或技术决策较多。                                      |
| - reviewer/后续维护者需要知道为什么这么做。                 |
|                                                            |
| 不需要，如果：                                               |
| - 实现较短。                                                |
| - specs 已足够表达上下文。                                  |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 3. 拆实现步骤                                                |
| 每一步必须能映射到：                                        |
| - PRODUCT.md 的行为编号，或                                  |
| - TECH.md 的 proposed change / validation item。             |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 4. 实现代码                                                  |
| - 遵循 TECH.md 的模块边界。                                  |
| - 遵循现有 repo patterns。                                  |
| - 添加必要测试或验证 artifact。                             |
| - specs、代码、测试尽量同一个 PR。                           |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 判断：实现是否改变了规格？                                  |
|                                                            |
| 更新 PRODUCT.md，如果：                                     |
| - 用户可见行为改变。                                        |
| - UX 细节、edge case、成功条件改变。                         |
|                                                            |
| 更新 TECH.md，如果：                                        |
| - 模块边界改变。                                            |
| - 状态、API、数据流、实现顺序改变。                         |
| - 风险或验证策略改变。                                      |
|                                                            |
| 不更新 specs，如果：                                        |
| - 只是内部等价重构。                                        |
| - 不影响规格承诺和验证计划。                                |
+-----------------------------+------------------------------+
                              |
                              v
+------------------------------------------------------------+
| 5. 验证                                                      |
| - rust-unit-tests：crate-level logic/regression。            |
| - warp-integration-test：重要用户 flow/P0 行为。             |
| - manual screenshots/videos：UI-heavy 或视觉状态。           |
| - 最终检查实现是否满足当前 specs，而不是旧草案。             |
+------------------------------------------------------------+
```

解释：原 skill：[implement-specs](https://github.com/warpdotdev/warp/blob/master/.agents/skills/implement-specs/SKILL.md)。

要点：
- 适用场景：`PRODUCT.md` 和必要的 `TECH.md` 已经批准，下一步是落地实现。
- 核心判断：`PRODUCT.md` 是用户行为真相，`TECH.md` 是实现计划真相；实现过程中发现行为或架构需要调整时，应同步更新 specs，而不是让文档过期。
- 关键产物：符合 specs 的代码、测试或验证 artifact、必要的 `PRODUCT.md`/`TECH.md` 更新，以及同一 PR 中连贯的实现叙事。
- 容易踩的坑：把 specs 当成一次性草稿、实现偏离后不更新文档、只验证代码能跑但不逐项映射回 Behavior invariants。
